## Genius Patterns

### 1. The Under-Specified Plan Diagnosis
- **What They Do Unconsciously**: Identifies that every failed vibe-coded project traces back to the same root cause — the developer didn't answer enough questions before prompting, forcing the LLM to make up the answers at generation time.
- **Executable Behavior**: Before any build, audit your brief for "unanswered questions." Every gap = a decision the LLM will make poorly. Fill gaps with research, not imagination.
- **Deployment Context**: Any AI-assisted development — landing pages, apps, components, full sites.
- **Success Metric**: Zero unanswered questions about audience, positioning, structure, or visual direction before first code prompt.

### 2. The Research → Design → Build Trinity
- **What They Do Unconsciously**: Never collapses the pipeline. Treats research, design, and implementation as three *categorical* steps with different tools, different outputs, and zero blending.
- **Executable Behavior**: Enforce a hard phase gate: Research produces competitive intelligence + section copy. Design produces approved visual mockups. Build produces code. Never skip or merge phases.
- **Deployment Context**: Any visual asset — landing pages, web apps, portfolio sites, lead magnets.
- **Success Metric**: Each phase has a discrete, reviewable deliverable before moving to the next.

### 3. Direct vs. Indirect Competitor Mapping
- **What They Do Unconsciously**: Expands the competitive research frame beyond obvious competitors to include *alternative solutions* to the same problem.
- **Executable Behavior**: For any product/service, identify (a) companies positioned around the same thing (direct), AND (b) entirely different approaches people use to solve the same underlying problem (indirect). The indirect competitors often reveal more valuable positioning insights.
- **Deployment Context**: Landing page copy, positioning strategy, unique selling proposition development.
- **Success Metric**: At least 2 direct + 2 indirect competitors mapped, with language, USP, and page structure captured for each.

### 4. The Competing Hypotheses Technique
- **What They Do Unconsciously**: Refuses to lock into a single positioning angle. Instead, generates multiple distinct approaches and evaluates them before committing.
- **Executable Behavior**: After research, generate 3 competing hypotheses for how to position the product — e.g., efficiency-focused, quality-focused, personalization-focused. Evaluate each against the audience before selecting.
- **Deployment Context**: Landing page copy, ad campaigns, product messaging, brand positioning.
- **Success Metric**: 3 distinct positioning angles generated and consciously evaluated (not just the first idea that sounds good).

### 5. Design Quality as Subconscious Authority
- **What They Do Unconsciously**: Treats visual quality not as aesthetic preference but as a *conversion mechanism* — people are drawn to authority, and design is a subconscious authority signal.
- **Executable Behavior**: Invest in visual polish proportional to the business impact. A beautiful landing page isn't about vanity — it's a trust signal that converts. Treat design as a revenue lever.
- **Deployment Context**: Any customer-facing asset — websites, pitch decks, product UIs, lead magnets.
- **Success Metric**: The visual quality creates an immediate impression of authority and professionalism before any copy is read.

### 6. The Structure-Before-Copy Sequencing (Counterintuitive)
- **What They Do Unconsciously**: Designs the visual scaffolding *without* real copy first, then layers in the researched copy as a second pass — counterintuitively producing better designs than doing both simultaneously.
- **Executable Behavior**: In design tools (Stitch, Figma, etc.), first nail the layout, typography, color palette, and visual rhythm with placeholder text. THEN swap in the real copy. This prevents the copy from constraining the design.
- **Deployment Context**: Landing page design, marketing sites, product pages.
- **Success Metric**: The design direction is approved before any real copy is introduced, preventing "copy-driven design" that produces mid-tier visuals.

### 7. The MCP Bridge Pattern
- **What They Do Unconsciously**: Uses MCP servers as the bridge between design tools and development environments, eliminating the "lossy handoff" where designs degrade during implementation.
- **Executable Behavior**: Connect design tools (Google Stitch) directly to development IDEs via MCP servers + skill libraries. The build reads the design system directly rather than interpreting screenshots.
- **Deployment Context**: Any design-to-code workflow, especially landing pages and marketing sites.
- **Success Metric**: The built output matches the approved design with zero manual CSS translation.

## Hidden Knowledge

### 1. The 99% Skip
99% of people skip competitive research. This is the #1 reason landing pages fail — not bad code, not ugly design, but speaking to no one in particular because you never researched who you're speaking to.

### 2. Landing Pages Are Solved Architecture
Landing pages follow the same structure virtually every time: hero (headline + subheadline + CTA) → social proof → features/benefits → how it works → pricing → FAQ. The value isn't in reinventing structure — it's in how precisely you fill each section with language that matches a specific person.

### 3. Stitch's Design Quality Inversely Correlates with Copy Completeness
When you give Stitch all the copy upfront, designs degrade to "mid." When you let it design the structure/aesthetic first and add copy second, designs are significantly better. The tool optimizes layout differently when it's not trying to accommodate specific text.

### 4. Style Keywords as Design DNA
A single design keyword (e.g., "claymorphism," "glassmorphism," "brutalist") can completely transform the output aesthetic. These keywords function as compressed design systems — they encode typography, color theory, spacing, and visual rhythm in a single word.

### 5. Image-as-Style-Transfer
Pasting a single reference image (like a video thumbnail) tells the design tool more about desired aesthetic than paragraphs of description. The tool extracts color palette, vibe, texture, and composition — producing outputs with coherent "design DNA" that text descriptions can't achieve.

## Methodology

### Phase 1: Competitive Intelligence Gathering

**Internal Research**: Document what you already know about your product, positioning, and audience. Write down the benefits, the target persona, and your initial positioning hypotheses.

**External Research (Direct Competitors)**: Find companies that position around the same value proposition. Extract their headline language, USP, brand identity, and page structure.

**External Research (Indirect Competitors)**: Find alternative solutions people use for the same underlying problem. These reveal blind spots in your positioning.

**Synthesis**: Generate 3 competing positioning hypotheses. Map each to a specific section structure for the landing page.

### Phase 2: Design Generation

**Scaffold Pass**: Generate the visual layout, typography, color palette, and section structure WITHOUT real copy. Get approval on the "bones."

**Copy Integration Pass**: Layer in the researched copy (headlines, subheadlines, CTAs, feature descriptions) into the approved design scaffolding.

**Style Variation**: Optionally test different design DNA (claymorphism, brutalism, etc.) or use reference images for style transfer.

### Phase 3: Code Build

**Tool Connection**: Set up MCP server for the design tool (Google Stitch) in the IDE.

**Skill Installation**: Install the design tool's skill library so the LLM understands the design system.

**One-Shot Build**: Issue a single command referencing the design project. The skills extract the design system and build the full page.

**Polish**: Iterate on bugs and fine-tune sections as needed.
