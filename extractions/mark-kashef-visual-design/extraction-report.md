## Executive Summary

- **Core Genius**: Using ASCII art wireframes as a "visual contract" between human intent and AI execution — forcing alignment on layout, structure, and spatial relationships BEFORE any code or design work begins. This eliminates the most expensive failure mode in AI-augmented design: the assumption gap.
- **What Makes This Different**: This isn't prompting technique or design skill — it's a **cognitive protocol**. It inserts a visual checkpoint between human vision and AI execution, turning vague mental models into precise spatial specifications that any AI tool can interpret perfectly.
- **Deployable Skills**: Blueprint any visual asset (websites, dashboards, slide decks, PDFs, marketing materials, database schemas) with spatial precision before committing to production. Iterate cheaply at the wireframe layer. Reduce token waste by 60-80% on visual projects.
- **Hidden Knowledge Captured**: The "taste arbitrage" — humans provide visual judgment and spatial reasoning while AI handles execution beauty. The thinking/beauty split. The token economics of pre-visualization.

## Genius Patterns

### 1. The Visual Contract Protocol
- **What He Does Unconsciously**: Forces every visual project through a blueprint phase where BOTH human and AI agree on spatial layout before any production work begins
- **Executable Behavior**: Before any design/code work, produce an ASCII wireframe. Iterate on the wireframe until it matches your mental model. THEN execute.
- **Deployment Context**: Any visual output — websites, dashboards, presentations, PDFs, marketing assets, email templates
- **Success Metric**: First production output matches intent with ≤1 revision cycle (vs. 5-6 without)

### 2. The Assumption Assassin
- **What He Does Unconsciously**: Identifies that AI failure in visual work almost always stems from **unstated assumptions**, not capability gaps. The wireframe eliminates assumptions by making spatial relationships explicit.
- **Executable Behavior**: Before prompting any visual build, list what the AI will assume. Then make each assumption explicit in the wireframe.
- **Deployment Context**: Complex layouts with multiple components that could be arranged in many valid configurations
- **Success Metric**: Zero "that's not what I meant" moments after wireframe approval

### 3. The Token Economist
- **What He Does Unconsciously**: Calculates the real cost of iterating at the code/design layer vs. the wireframe layer. Slide decks especially — "very token intensive" — 5-6 iterations at the code layer can exhaust context windows entirely.
- **Executable Behavior**: Route ALL iteration to the wireframe layer where changes cost ~50 tokens vs. ~5,000 tokens at the code layer. Only "commit to production" once the wireframe is locked.
- **Deployment Context**: Token-expensive outputs (slide decks, multi-page layouts, complex dashboards), long sessions, context-window-sensitive work
- **Success Metric**: Token spend per project reduced by 60-80%

### 4. The Taste Arbitrage
- **What He Does Unconsciously**: Separates the "thinking" (layout, structure, relationships, hierarchy) from the "beauty" (colors, typography, spacing, polish). Humans handle taste — AI handles execution.
- **Executable Behavior**: In the wireframe, specify WHAT goes WHERE and WHY. In the build prompt, tell AI to focus on making it beautiful. Don't mix the two cognitive tasks.
- **Deployment Context**: Any design work where aesthetic quality matters alongside structural correctness
- **Success Metric**: Output is both structurally correct AND aesthetically excellent — not one at the expense of the other

### 5. The Progressive Refinement Engine
- **What He Does Unconsciously**: Uses multi-turn conversation at the wireframe layer to progressively refine, making small targeted changes ("two changes only") rather than holistic rewrites
- **Executable Behavior**: After initial wireframe, make scoped, numbered edits. "Change X in section Y. Add Z between A and B. Redraw." Each edit is surgical.
- **Deployment Context**: Complex layouts with many interdependent components
- **Success Metric**: Each refinement cycle takes <30 seconds and moves measurably closer to intent

### 6. The Complexity Equalizer
- **What He Does Unconsciously**: Uses visual diagrams to make technical concepts (database schemas, system architecture, SQL relationships) accessible to non-technical stakeholders
- **Executable Behavior**: For any technical system, request an ASCII visualization before implementation. If it "looks like gibberish," ask for simplification ("make this as if you're in seventh grade").
- **Deployment Context**: Database design, system architecture, API relationships, data flows, any technical planning for non-technical decision-makers
- **Success Metric**: Non-technical person can look at the diagram and say "yes, that's what I want" or "no, we need X"

### 7. The Multi-Vertical Blueprint System
- **What He Does Unconsciously**: Applies the SAME visual planning protocol across radically different output types — dashboards, landing pages, slide decks, databases — proving it's a universal cognitive protocol, not a domain-specific trick
- **Executable Behavior**: Memorize the pattern: "Before writing any [output type], create an ASCII wireframe of [specifications]." Apply to ANY visual output.
- **Deployment Context**: Universal — any project with a visual or structural component
- **Success Metric**: Protocol works identically across 5+ output types

### 8. The Orchestrator Identity Shift
- **What He Does Unconsciously**: Repositions the human from "prompt engineer" to "creative director" — you sketch the blueprint, you approve the wireframe, you greenlight production. The AI is your production team.
- **Executable Behavior**: Adopt the conductor mindset. Your job is vision, taste, and approval gates. AI's job is execution at the production layer.
- **Deployment Context**: Any AI-augmented creative workflow
- **Success Metric**: You feel like a director reviewing work, not a coder debugging output

## Methodology

### Phase 1: Conceptualize
Define what you want to build. Identify components, relationships, and hierarchy. Don't touch code, don't touch design tools, don't prompt for production output.

### Phase 2: Wireframe
"Before writing any [code/slides/design], create an ASCII wireframe of [specification]." Include spatial relationships, component labels, hierarchy clues.

### Phase 3: Iterate
Make small, numbered, scoped changes to the wireframe. "Two changes only." Keep iterating until wire matches mental model perfectly.

### Phase 4: Annotate
Add style/aesthetic annotations to the locked wireframe. Color hints, typography preferences, icon quality standards ("no vibecoded icons").

### Phase 5: Execute
"Build this as a [React app / slide deck / landing page / etc.] using the wireframe as the exact specification." AI focuses on beauty; structure is already solved.

### Phase 6: Validate
Compare output to wireframe. Every element should map 1:1 to the specification.
