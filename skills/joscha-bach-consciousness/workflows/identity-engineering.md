# Identity Engineering

> Bach's radical reframe: identities are tools, not truths. "I need to be identified to get things done." This workflow lets you construct, deploy, and release identities without existential crisis.

## When to Use

- Designing a personal brand or professional identity
- Stuck in identity fusion — can't separate yourself from a role or outcome
- Need to adopt a new professional persona for a project
- Someone (or you) is defending an identity instead of using it
- Building a character for content, fiction, or roleplay

## The Identity Engineering Protocol

### Phase 1: Identity Audit

Map your current identities:

| Identity | Function (what it lets me do) | Active? | Still useful? |
|----------|------------------------------|---------|---------------|
| [role 1] | [what tasks this identity enables] | Y/N | Y/N |
| [role 2] | [what tasks this identity enables] | Y/N | Y/N |

**Key Question** (from Bach): "Is this identity the right instrument for the task in front of me?"

Flag identities that are:
- **Fused**: You can't separate yourself from this role (dangerous)
- **Vestigial**: This identity served a past purpose but you're still carrying it
- **Aspirational**: This identity is what you want to become but haven't built yet

### Phase 2: Identity Construction

For each identity you need to build or rebuild:

**1. Functional Specification**
- What tasks does this identity need to enable?
- What decisions does this identity make?
- What does this identity say "no" to?

**2. Persistence Conditions** (from the Spirit = Software pattern)
- What substrates does this identity operate on? (in-person, online, writing, video)
- What would kill this identity? (inconsistency, contradiction, context collapse)
- What keeps this identity alive? (rituals, triggers, environments)

**3. Ignition Conditions** (from the Phase Transition Detector)
- What must happen for someone (including you) to recognize this identity as real?
- Not gradual — what's the ignition point?
- How many interactions before the identity coheres?

### Phase 3: Identity Deployment

For the active session/project/task:

1. **Select**: Which identity is the right instrument for this task?
2. **Activate**: What's the entry ritual? (Bach doesn't say this explicitly but his framework implies it — identities have activation conditions)
3. **Operate**: What does this identity do differently than "default you"?
4. **Monitor**: Is the identity serving the task, or has the task started serving the identity? (Identity fusion check)

### Phase 4: Identity Release

**Bach's insight**: "We arrive as passengers in our identity."

When a task is done, the identity should be releasable:

1. **Deactivate**: Explicitly end the identity session. "I was operating as [X]. That task is complete."
2. **Archive**: Store what this identity learned for future deployment
3. **Check**: Can you put this identity down without existential anxiety? If not, you have fusion — return to Phase 1

## Output Format

```
IDENTITY: [Name/role]

FUNCTION: [What tasks it enables]
PERSISTENCE: [What substrates it lives on, what kills it, what sustains it]
IGNITION: [Phase transition point — when does it become "real"?]

DEPLOYMENT TRIGGERS: [When to activate this identity]
RELEASE PROTOCOL: [How to deactivate cleanly]

FUSION RISK: [Low/Medium/High — how likely is identity fusion?]
FUSION MITIGATION: [What prevents fusion]
```

## Quality Gate

- Can you articulate what this identity DOES separately from what you ARE?
- Is there a clear entry and exit for identity deployment?
- Have you identified fusion risk and mitigation?
- Does this identity serve the task, or has the task started serving the identity?
