# Identity Version Builder

Design and deploy your next identity iteration — the written spec sheet that becomes your operating system.

---

## Context

Jeremy Haynes treats identity as firmware that gets flashed with explicit updates. He's on version 47.0. Each version is a written document: a list of traits and characteristics posted visibly and referenced during live situations. The shift is a light switch — you decide, you document, you start acting through the new traits immediately. Growth comes from the acting, not from preparing to act.

**Source Pattern:** Genius Pattern #1 — Identity Versioning Protocol

---

## Prompt

```
You are an identity engineer using Jeremy Haynes' Identity Versioning Protocol.

I need to build my next identity version. Here's my context:

**Current situation:** [Describe where you are — revenue, role, lifestyle, daily reality]
**Current identity version traits I'm aware of:** [List traits you currently operate through — how you show up, what you believe about yourself, habits, patterns]
**What's blocking me:** [The specific behavior, belief, or pattern that's creating a ceiling]
**Where I want to be in 90 days:** [Concrete outcome — revenue, lifestyle, capability]

---

Using Jeremy Haynes' Identity Versioning Protocol, build my next version:

### STEP 1: VERSION AUDIT
Analyze my current traits. Identify which ones served me to get HERE but are now creating the ceiling. Name the specific trait-to-blocker connection (e.g., "Your trait of 'doing everything yourself' got you to $500K but is the exact thing preventing $2M because it caps your leverage").

### STEP 2: VERSION DESIGN
Create my next identity version document:

**[Name] Version [X].0**

List 8-12 traits and characteristics for the new version. Rules:
- Most traits carry forward from the current version (identity versioning changes as few as ONE trait per iteration)
- Changed/new traits must be specific and behavioral, not aspirational platitudes
- Each trait should be something I can reference mid-conversation or mid-decision as a real-time course-correction
- Format each as: "[TRAIT]: [One-sentence operating definition]"

Mark which traits are NEW, which are MODIFIED, and which CARRY FORWARD.

### STEP 3: DEPLOYMENT PROTOCOL
1. **The Document** — Exactly what to write, where to post it, how to format it for glance-ability
2. **The Light Switch Moment** — The specific decision/declaration that activates this version (not a gradual transition)
3. **Live Reference Points** — 3 specific situations in my daily life where I should glance at this document and consciously act through the new traits
4. **The Test** — One situation in the next 7 days that will test the new version. Describe exactly how old-version-me would handle it vs. new-version-me.

### STEP 4: VERSION CHANGELOG
Write a brief changelog entry:
- **Version [X-1].0 → [X].0**
- **Primary change:** [The core shift]
- **Why now:** [What made this version necessary]
- **Expected unlock:** [What becomes possible that wasn't before]

Remember: "Every trait and characteristic of you is fully fluid and dynamic. 100% of what you consider you can change." This is not about becoming someone new — it's about updating the spec sheet you operate through.
```

---

## Quality Check

Output passes if:
- [ ] Traits are behavioral and specific (not "be more confident" but "default to decisive action within 60 seconds of receiving information")
- [ ] Changed traits have clear connection to the identified ceiling
- [ ] The deployment protocol is actionable TODAY, not someday
- [ ] The light switch moment is a decision, not a process
- [ ] The test scenario makes the difference between versions concrete and visceral
